# 如何设计一个能连续执行 2 小时任务的 Coding Agent

> Kimi Product Engineer 笔试题 1。
> 这不是一篇纯设计文档——我把四个机制都实现成了可开关的原型，并用故障注入评测量化了每个机制的价值。
> 代码：本仓库；调研：[research.md](research.md)；下面每个数字都能用 `python -m evals.scenarios` 复现。

## 结论先行

「2 小时」这个时长本身就是题眼。它把三件在短任务里**偶发**的事变成了**必然**：

1. **上下文必然溢出**——几百轮工具调用累积的日志会撑爆任何窗口。
2. **故障必然发生**——网络抖动、429、5xx、命令挂死，在 2 小时尺度上是统计必然。
3. **用户必然要插手**——没人能写出一句话指令让 agent 无偏差自主跑 2 小时；中途一定要改方向。

所以这四个机制不是「优化项」，是长任务能不能跑完的**必需品**。我的设计原则一以贯之：

> **状态进磁盘，故障能分类，中断有安全点，上下文按价值取舍。**

四个机制对应四个模块，每个都能用一个 `--no-xxx` 开关关掉，于是「它到底有没有用」不靠嘴说，靠对照实验：

| 机制 | 关掉它会怎样（本仓库 Kimi 实测） |
|---|---|
| ① 上下文管理 | 输入 token 单调膨胀（2726 且不回落 vs 开启后压回 1263），长任务必然撞窗口 |
| ② 任务状态 | 进程一崩，崩溃前的工作全清零，只能从头再跑（实测崩溃后 resume 几乎零重复，关掉则 11 轮重付） |
| ③ 失败恢复 | 第一个瞬时 API 故障（注入的 502）就让整个任务在第 0 轮退出 |
| ④ 用户中断 | 要么干等到底，要么 kill 进程丢进度，无法中途把 Hello 改成 Goodbye |

## 机制① 上下文管理

**问题**：长任务的上下文是单调增长的。每轮工具调用都往 messages 里塞进新的命令输出、文件内容。
线性堆下去，输入侧 token 暴涨，最终要么撞窗口报错，要么被 provider 静默截断（更糟，模型不知道丢了什么）。

**我的设计：三层防线**（[harness/context.py](../harness/context.py)）

1. **工具结果截断 + 落盘（offload）**：单条工具结果超 4000 字符，就只保留头尾、中间留标记，
   完整内容写到 `.agent/outputs/`，并告诉模型「完整结果在这个文件，需要时 read_file」。
   大块低价值文本不进上下文，但没真正丢——模型可按需取回。
2. **历史压缩（compaction）**：当输入 token ≥ 窗口 × 75%，调一次 LLM 把旧历史压成结构化摘要
   （当前焦点 / 已完成 / 错误与解法 / 关键文件 / 设计决定 / TODO），替换被压缩段，保留最近 3 个完整 turn。
   **切割铁律**：切点只落在完整 turn 边界，绝不把 `tool_call` 和它的 `tool_result` 切散——切散直接 400。
3. **文件系统即记忆**：真正要紧的状态（todo.md、产物代码）本就在 workspace 文件里，不依赖对话记忆。
   压缩丢掉的是过程，不是结果。

**为什么是 75% 而不是撑满**：压缩本身要调一次 LLM、要占输出 token，必须在撞墙前留出余量做这件事。
双条件触发（占比 75% 或剩余空间不足）取「先到者」，是 Kimi CLI 的设计——比单一阈值更稳。

**主流对照**（详见 research.md）：
- Kimi CLI：双条件触发（用量 ≥85% 或剩余 <50k），LLM 摘要替换历史，压缩提示词按优先级排序（当前任务 > 错误 > 代码最终态 > …）。我直接借鉴了它的优先级设计。
- Claude Code：auto-compact + microcompact 双层，UI 常驻「离 auto-compact 还有多远」，工具结果超 40k 截断。我的 offload 是它的简化版。
- pi：按 turn 边界切割，reserveTokens + keepRecentTokens。我的「切割铁律」来自它。

**实验数据**（S1，小窗口 3000 强制触发压缩）：

<!-- DATA_S1（model=kimi-for-coding，复现自 evals/results/summary_kimi.csv）-->
| 变体 | 完成 | 轮数 | 峰值输入 token | 末轮输入 token | 压缩次数 |
|---|---|---|---|---|---|
| compact_on  | 是 | 19 | **2259** | **1263** | 1 |
| compact_off | 是 | 20 | **2726** | **2726** | 0 |

读法：两者跑同一个「建 5 个文件逐个验证」的任务。
- **compact_on**：第 1 次压缩在第 ~14 轮触发，把上下文从 2259 压到 1263——末轮输入 token **低于**峰值，曲线是锯齿状、有界的。
- **compact_off**：输入 token 单调爬升到 2726 且**不回落**（峰值=末值）。

窗口被我调到 3000 只为让压缩在十几轮内触发；在真实 256k 窗口的 2 小时任务里，off 那条单调曲线终会撞墙报错。
这就是「关掉机制① → 长任务必然崩」的可视化——而开启后上下文始终被摁在窗口内。

## 机制② 任务状态记录

**问题**：状态只活在内存里，一次崩溃（OOM、断电、被 kill）就抹掉 2 小时工作。长任务必须假设「进程随时会死」。

**我的设计：物理层 + 语义层**（[harness/state.py](../harness/state.py)）

- **物理层 = append-only JSONL**：每个事件（user/assistant/tool_result/round/compaction）一发生就
  `write + flush + fsync` 落盘。这是会话的**唯一权威记录**，人类可读，`jq` 可审计。
  resume 的本质是**重放这个文件把 messages 拼回来，从断点继续**——不是重跑。
- **语义层 = todo.md**：模型自己维护的 `- [ ]`/`- [x]` 清单，写在 workspace 里（不受上下文压缩影响）。
  resume 后模型先读它，一眼看出「哪些做完了」，避免重做。

**为什么两层都要**：物理层保证「消息不丢」，但重放出一堆消息后，模型仍需要一个**人类语义**的进度锚点——
todo.md 就是那个锚点。物理层是给机器 replay 的，语义层是给模型「认路」的。

**主流对照**：
- Codex：rollout JSONL（`~/.codex/sessions/rollout-*.jsonl`），append + 显式 flush，可重放。我的物理层与之同构。
- Claude Code：会话 JSONL + TodoWrite 清单；resume 时注入「本会话由先前会话续接」标记。我的语义层与续接标记都来自它。
- MiMo Code（小米）：更进一步，用 SQLite 存原始轨迹（权威）+ checkpoint.md/MEMORY.md 记忆文件（缓存），
  甚至有 `/dream` 离线把多会话记忆固化。这是「跨会话持久记忆」，比单会话 resume 高一个量级——
  我在原型里没做（见末尾取舍），但它指出了状态机制的演进方向。

**实验数据**（S3，跑到第 2 轮真 kill -9，再 resume）：

<!-- DATA_S3（model=kimi-for-coding）-->
| 变体 | 完成 | 总轮数 | 说明 |
|---|---|---|---|
| resume_on | 是 | 12 | 第 2 轮后 `os._exit(137)` 真杀进程（崩溃前已 fsync **9 条事件**）；resume 子进程重放后从第 3 轮续到第 12 轮完成 |
| no_resume(baseline) | 是 | 11 | 同一任务从头一口气跑完需 11 轮——崩溃后若不 resume，这 11 轮要**全部重付** |

读法（这里有个容易看反的点，要讲清）：崩溃发生在第 2 轮、进程被 `kill -9` 级别地真实杀死。
因为 events.jsonl 逐行 `fsync`，崩溃前的 2 轮工作（含已建好的文件、已打勾的 todo.md）**留在磁盘上没丢**。
resume 重放这些事件、读 todo.md 认出做到哪了，**从第 3 轮继续而非第 1 轮**，没有重做任何已完成步骤。

算一笔账：resume 路径总代价 = 崩溃前 2 轮 + resume 增量 10 轮 ≈ 基线一次跑完的 11 轮，**几乎零重复劳动**。
而关掉机制②（不 resume）的代价 = 崩溃前 2 轮直接蒸发 + 重新从头跑 11 轮。任务越长、崩溃越靠后，这个差距越致命——
一个跑到第 90 轮才崩的 2 小时任务，没有机制②就是 90 轮工作清零。

## 机制③ 工具调用失败恢复

**问题**：2 小时里瞬时故障是必然，但不能无脑重试（参数错了重试一万次还是错）。核心是**分类**。

**我的设计：分层 + 分类**（[harness/recovery.py](../harness/recovery.py)）

- **API 层**（`with_retry`）：
  - 429 / 5xx / 超时 / 连接错 → 可恢复，**指数退避 + jitter**（200ms × 2ⁿ，jitter 0.9–1.1，最多 5 次）
  - 4xx 参数错（400/401/404）→ 不可恢复，直接抛，重试无意义
  - 上下文溢出 → 特殊信号 `ContextOverflow`，主循环先做一次紧急压缩再重试
- **工具层**（主循环）：工具抛任何异常都被捕获，转成错误文本作为 tool result **回喂模型让它自纠**，绝不冒泡崩主循环。
- **防呆**（`LoopGuard`）：相同工具+相同参数连续 3 次 → 判死循环，注入纠正提示；连续失败 5 次 → 熔断，让模型停下重新评估。

**为什么退避要 jitter**：多个请求同时被限流后，若都按同一节奏重试会形成「重试风暴」再次打爆服务端。
随机抖动把重试时间打散。这个 200ms×2ⁿ + 0.9–1.1 的参数我直接取自 Codex 的实现。

**为什么工具错误要回喂而不是重试**：工具失败往往不是瞬时的（文件不存在、命令语法错），
重试同样的调用没意义——把错误给模型看，让它换个做法，才是恢复。这是业界共识。

**主流对照**：
- Codex：退避 200ms×2ⁿ + jitter 0.9–1.1（我的参数来源）。
- Kimi CLI：工具错误类型化（NotFound/Parse/Validate/Runtime）回喂；调用去重防死循环；
  **官方自带 ChaosChatProvider 故障注入测试**——故障恢复要测出来，不是写了就算。我的 [evals/chaos.py](../evals/chaos.py) 与之同构。

**实验数据**（S2，第一次 API 调用必失败 + 后续 25% 概率失败，seed 固定可复现）：

<!-- DATA_S2（model=kimi-for-coding，chaos seed=7：首次调用必失败 + 后续 25% 概率失败）-->
| 变体 | 完成 | 轮数 | 说明 |
|---|---|---|---|
| retry_on | **是** | 7 | 退避重试 **6 次**后跑完整个任务 |
| retry_off | **否** | 0 | 撞上第一个注入的 **502** 直接退出，任务在第 0 轮就死 |

读法：retry_off 撞上第一个注入的故障就整任务退出（done=False）；retry_on 退避重试后跑完。
同一个 seed、同一个任务，唯一变量是重试开关——这是机制③价值最干净的一刀切对照。

**附 S4（工具失败自纠）**：任务里埋一条必失败命令（`python nonexistent_xyz.py`），观察模型能否从错误里恢复。

<!-- DATA_S4 -->
| 变体 | 完成 | tool_fails | 说明 |
|---|---|---|---|
| self_correct | **是** | 1 | 必失败命令触发 1 次工具失败；错误被回喂后模型改建 hello.py 并成功运行完成 |

读法：tool_fails=1 但 completed=True，说明工具失败没有崩主循环，而是作为错误文本回喂、被模型自纠掉了。

## 机制④ 用户中断

**问题**：没人能写一句话让 agent 无偏差跑 2 小时。中途一定要插话改向。如果中断只能靠 kill 进程，
用户要么干等，要么丢全部进度。中断必须是一等公民，且满足两个性质：

**我的设计：安全点 + 中断即上下文**（[harness/interrupt.py](../harness/interrupt.py)）

1. **安全点**：后台线程非阻塞监听 stdin，用户运行中敲的字进队列，
   只在**工具调用边界**这个安全点取出注入——不会打断一次正在进行的文件写入或状态落盘。
   任何时刻停下，磁盘状态都是自洽的。
2. **中断即上下文**：用户打断不是悄悄丢弃，而是作为一条 user 消息注入对话——
   模型恢复时看得见「我在做 X 时被要求改成 Y」，据此改向而非懵掉。
3. **两级 Ctrl-C**：一次 = 优雅停（落盘 + 打印 resume 命令），二次 = 强退（append-only 已保证不丢）。
4. **危险命令审批门**：`run_command` 命中 `rm -rf`/`git push`/`sudo` 等模式时暂停等用户确认（非交互环境默认拒绝）。

**为什么安全点是工具边界**：这是「哪些操作可被取消、哪些必须原子完成」的显式划分。
LLM 调用、工具执行可以被打断；但消息列表的增长、状态落盘不可以——它们必须原子完成，否则会留下半写的脏状态。
这个思想直接来自 Kimi CLI 用 `asyncio.shield` 保护 context 写入的设计。

**主流对照**：
- Kimi CLI：可中断的 step/工具 + `asyncio.shield` 保护的 context 写入 = 安全点。我用同步方式复刻了这个性质。
- Claude Code：Esc 中断注入「[Request interrupted by user]」进对话，模型知道自己被打断在哪。我的「中断即上下文」来自它。
- Codex：协议层 `Op::Interrupt` → `TurnAborted`，中断是一等操作而非信号 hack。

**实验数据**（S5，第 2 轮注入「改打印 Goodbye」）：

<!-- DATA_S5（model=kimi-for-coding）-->
| 变体 | 完成 | 产物 greet.py 内容 | 是否采纳新需求 |
|---|---|---|---|
| steer_on | 是 | 打印 **Goodbye** | ✅ 采纳 |
| steer_off(baseline) | 是 | 打印 **Hello** | —（未注入） |

读法：任务本是「让 greet.py 打印 Hello」。steer_on 在第 2 轮被注入「改成 Goodbye」后，最终产物打印 Goodbye；
steer_off 不注入则保持 Hello。同一任务、唯一变量是中途那句插话——产物从 Hello 变成 Goodbye，
证明插话确实在安全点改变了 agent 的行为轨迹，而不是被忽略或覆盖原始指令。

## 评测方法论

- **可复现**：故障注入带 seed（[evals/chaos.py](../evals/chaos.py)），同 seed 同结果。
- **控制变量**：每个机制单独开/关，其余不变，差异可归因到该机制。
- **真实而非 mock**：S3 用 `os._exit(137)` 真实杀进程、子进程 resume，不是假装崩溃。
- **指标**：完成与否（脚本断言）、轮数、输入 token 峰值/末值、压缩/重试/工具失败次数。
- **双模型**：同一套评测在 Kimi For Coding 与 DeepSeek 上各跑一遍，对比两家在长任务可靠性上的表现差异本身就是观察。

完整结果：[evals/results/](../evals/results/)。

## 边界与取舍（没做什么，为什么）

诚实地说，原型刻意没做这些，因为它们超出「单 agent 跑 2 小时」的核心：

- **跨会话持久记忆 / 自进化**：MiMo Code 的 SQLite 轨迹库 + `/dream` 记忆固化是更高阶的状态机制，
  但那解决的是「跨任务复用经验」，不是「单个长任务跑完」。我在机制②点明了它是演进方向，但原型聚焦单会话。
- **沙箱隔离**：真实生产要把 `run_command` 关进容器/seccomp。我用 workspace 路径钉死 + 危险命令审批门做了最小约束，
  但没上沙箱——那是另一个完整工程，且与四机制正交。
- **Multi-Agent / 子代理编排**：长任务可以拆给子代理并行，但那会引入协调、上下文隔离的新复杂度，
  对「先把单 agent 跑稳」这个目标是过早优化。
- **流式输出 / 精细 UI**：原型用 print，够说明机制；真实产品的 TUI 是另一层工作。

这些不是「不知道」，是「这道题的核心是 harness 可靠性的四块基石，先把它们做扎实并量化，比铺开功能更有价值」。
